01 · Problem

为什么不是只写一段角色 Prompt

Prompt 可以快速验证角色方向，但当对话轮次增长、历史记忆变长或输出格式变复杂时，单纯依赖提示词容易出现稳定性下降，因此需要把数据、训练与评测放在同一套产品流程中。

01

角色一致性不足

模型在长对话中容易偏离人物语气、价值观和表达习惯，导致数字人“越聊越不像本人”。

02

上下文与记忆断裂

历史事实、关系状态和当前情绪不能稳定承接，容易重复提问、前后矛盾或忽略关键事件。

03

输出格式不稳定

业务需要的 [emoji]、[voice] 等结构化标记可能缺失、错位或被自然语言污染。

02 · Workflow

从历史对话到可调用模型

以可复现、可评测和可交付为目标，将模型微调拆成五个相互约束的阶段，而不是把“训练成功”当作项目终点。

STEP 01

数据治理

清洗、去重、脱敏、角色归属确认和低质量样本剔除。

STEP 02

数据结构化

统一为 JSONL，补充角色设定、上下文和输出约束字段。

STEP 03

训练验证

划分训练集与验证集，观察 Loss 与异常样本表现。

STEP 04

标准评测

用固定测试 Prompt 和评分卡对不同版本进行横向比较。

STEP 05

部署调用

通过 API 验证真实上下文、格式标记和业务集成流程。

03 · Product Design

Prompt 结构与评分卡

将角色信息、当前对话、长期记忆和格式要求拆成稳定字段，让模型训练、在线调用和问题定位使用同一种输入结构。

四层 Prompt 结构

角色设定身份、语气、关系、价值观、禁区与行为边界。
当前上下文本轮用户输入、当前任务和即时情绪线索。
历史记忆需要持续保持的人物关系、事件和用户偏好。
输出约束回答长度、表达风格以及 [emoji]、[voice] 等结构化标记。

04 · Boundaries

方案边界与风险控制

模型微调不是万能答案。项目同时关注数据授权、隐私脱敏、过拟合、幻觉和在线调用成本，避免只展示理想输出。

适合使用微调的情况

  • 角色风格需要长期稳定保持
  • 存在较高质量且可授权的数据
  • 固定输出格式需要更高合规率
  • Prompt 已验证但稳定性仍不足

必须持续监控的风险

  • 敏感数据泄露与未经授权训练
  • 小数据集过拟合和机械复读
  • 角色记忆与事实准确性混淆
  • 模型升级后评测口径不可比较
